19//! # Testing with `gtest`
20//!
21//! `gtest` simulates a real network by providing mockups of the user, program,
22//! balances, mailbox, etc. Since it does not include parts of the actual
23//! blockchain, it is fast and lightweight. But being a model of the blockchain
24//! network, `gtest` cannot be a complete reflection of the latter.
25//!
26//! As we said earlier, `gtest` is excellent for unit and integration testing.
27//! It is also helpful for debugging Gear program logic. Nothing other than the
28//! Rust compiler is required for running tests based on `gtest`. It is
29//! predictable and robust when used in continuous integration.
30//!
31//! ## Main concepts
32//!
33//! `gtest` is a library that provides a set of tools for testing Gear programs.
34//! The most important structures are:
35//!
36//! - [`System`] — a structure that represents the environment of the Gear
37//!   network. It contains the current block number, timestamp, and other
38//!   parameters. It also stores the mailbox and the list of programs.
39//! - [`Program`] — a structure that represents a Gear program. It contains the
40//!   information about program and allows sending messages to other programs.
41//! - [`Log`] — a structure that represents a message log. It allows checking
42//!   the result of the program execution.
43//!
44//! Let's take a closer look at how to write tests using `gtest`.
45//!
46//! ## Import `gtest` lib
47//!
48//! To use the `gtest` library, you must import it into your `Cargo.toml` file
49//! in the `[dev-dependencies]` block to fetch and compile it for tests only:
50//!
51//! ```toml
52//! [package]
53//! name = "my-gear-app"
54//! version = "0.1.0"
55//! authors = ["Your Name"]
56//! edition = "2024"
57//!
58//! [dependencies]
59//! gstd = { git = "https://github.com/gear-tech/gear.git", tag = "v1.0.1" }
60//!
61//! [build-dependencies]
62//! gear-wasm-builder = { git = "https://github.com/gear-tech/gear.git", tag = "v1.0.1" }
63//!
64//! [dev-dependencies]
65//! gtest = { git = "https://github.com/gear-tech/gear.git", tag = "v1.0.1" }
66//! ```
67//!
68//! ## Program example
69//!
70//! Let's write a simple program that will receive a message and reply to it.
71//!
72//! `lib.rs`:
73//!
74//! ```ignore
75//! #![no_std]
76//! use gstd::msg;
77//!
78//! #[unsafe(no_mangle)]
79//! extern "C" fn handle() {
80//!     let payload = msg::load_bytes().expect("Failed to load payload");
81//!
82//!     if payload == b"PING" {
83//!         msg::reply_bytes(b"PONG", 0).expect("Failed to send reply");
84//!     }
85//! }
86//! ```
87//!
88//! `build.rs`:
89//!
90//! ```ignore
91//! fn main() {
92//!     gear_wasm_builder::build();
93//! }
94//! ```
95//!
96//! We will add a test that will check the program's behavior. To do this, we
97//! will use the `gtest` library.
98//!
99//! Our test will consist of the following steps:
100//!
101//! 1. Initialize the `System` structure.
102//! 2. Initialize the `Program` structure.
103//! 3. Send an init message to the program. Even though we don't have the `init`
104//!    function in our program, the first message to the program sent via
105//!    `gtest` is always the init one.
106//! 4. Send a handle message to the program.
107//! 5. Check the result of the program execution.
108//!
109//! Add these lines to the bottom of the `lib.rs` file:
110//!
111//! ```no_run
112//! #[cfg(test)]
113//! mod tests {
114//!     use gtest::{Log, Program, System};
115//!
116//!     // Alternatively, you can use the default users from `gtest::constants`:
117//!     // `DEFAULT_USER_ALICE`, `DEFAULT_USER_BOB`, `DEFAULT_USER_CHARLIE`, `DEFAULT_USER_EVE`.
118//!     // The full list of default users can be obtained with `gtest::constants::default_users_list`.
119//!     const USER_ID: u64 = 100001;
120//!
121//!     #[test]
122//!     fn test_ping_pong() {
123//!         // Initialization of the common environment for running programs.
124//!         let sys = System::new();
125//!
126//!         // Initialization of the current program structure.
127//!         let prog = Program::current(&sys);
128//!
129//!         // Provide user with some balance.
130//!         sys.mint_to(USER_ID, EXISTENTIAL_DEPOSIT * 1000);
131//!
132//!         // Send an init message to the program.
133//!         let init_message_id = prog.send_bytes(USER_ID, b"Doesn't matter");
134//!
135//!         // Run execution of the block which will contain `init_message_id`
136//!         let block_run_result = sys.run_next_block();
137//!
138//!         // Check whether the program was initialized successfully.
139//!         assert!(block_run_result.succeed.contains(&init_message_id));
140//!
141//!         // Send a handle message to the program.
142//!         let handle_message_id = prog.send_bytes(USER_ID, b"PING");
143//!         let block_run_result = sys.run_next_block();
144//!
145//!         // Check the result of the program execution.
146//!         // 1. Create a log pattern with the expected result.
147//!         let log = Log::builder()
148//!             .source(prog.id())
149//!             .dest(USER_ID)
150//!             .payload_bytes(b"PONG");
151//!
152//!         // 2. Check whether the program was executed successfully.
153//!         assert!(block_run_result.succeed.contains(&handle_message_id));
154//!
155//!         // 3. Make sure the log entry is in the result.
156//!         assert!(block_run_result.contains(&log));
157//!     }
158//! }
159//! ```
160//!
161//! To run the test, use the following command:
162//!
163//! ```bash
164//! cargo test
165//! ```
166//!
167//! # `gtest` capabilities
168//!
169//! Let's take a closer look at the `gtest` capabilities.
170//!
171//! ## Initialization of the network environment for running programs
172//!
173//! ```no_run
174//! # use gtest::System;
175//! let sys = System::new();
176//! ```
177//!
178//! This emulates node's and chain's behavior. By default, the [`System::new`]
179//! function sets the following parameters:
180//!
181//! - current block equals `0`
182//! - current timestamp equals UNIX timestamp of your system
183//! - starting message id equals `0x010000..`
184//! - starting program id equals `0x010000..`
185//!
186//! ## Program initialization
187//!
188//! There are a few ways to initialize a program:
189//!
190//! - Initialize the current program using the [`Program::current`] function:
191//!
192//!     ```no_run
193//!     # use gtest::Program;
194//!     # let sys = gtest::System::new();
195//!     let prog = Program::current(&sys);
196//!     ```
197//!
198//! - Initialize a program from a Wasm-file with a default id using the
199//!   [`Program::from_file`] function:
200//!
201//!     ```no_run
202//!     # use gtest::Program;
203//!     # let sys = gtest::System::new();
204//!     let prog = Program::from_file(
205//!         &sys,
206//!         "./target/wasm32-gear/release/demo_ping.wasm",
207//!     );
208//!     ```
209//!
210//! - Initialize a program via builder:
211//!
212//!     ```no_run
213//!     # use gtest::ProgramBuilder;
214//!     # let sys = gtest::System::new();
215//!     let prog = ProgramBuilder::from_file("your_gear_program.wasm")
216//!         .with_id(105)
217//!         .build(&sys);
218//!     ```
219//!
220//!     Every place in this lib, where you need to specify some ids, it requires
221//!   generic type `ID`, which implements ``Into<ProgramIdWrapper>``.
222//!
223//!     `ProgramIdWrapper` may be built from:
224//!     - `u64`
225//!     - `[u8; 32]`
226//!     - `String`
227//!     - `&str`
228//!     - [`ActorId`](https://docs.gear.rs/gear_core/ids/struct.ActorId.html)
229//!       (from `gear_core` one's, not from `gstd`).
230//!
231//!     `String` implementation means the input as hex (with or without "0x").
232//!
233//! ## Getting the program from the system
234//!
235//! If you initialize program not in this scope, in cycle, in other conditions,
236//! where you didn't save the structure, you may get the object from the system
237//! by id.
238//!
239//! ```no_run
240//! # let sys = gtest::System::new();
241//! let prog = sys.get_program(105).unwrap();
242//! ```
243//!
244//! ## Initialization of styled `tracing-subscriber`
245//!
246//! Initialization of styled `tracing-subscriber` to
247//! print logs (only from gwasm` by default) into stdout:
248//!
249//! ```no_run
250//! # let sys = gtest::System::new();
251//! sys.init_logger();
252//! ```
253//!
254//! To specify printed logs, set the env variable `RUST_LOG`:
255//!
256//! ```bash
257//! RUST_LOG="target_1=logging_level,target_2=logging_level" cargo test
258//! ```
259//!
260//! ## Pre-requisites for sending a message
261//!
262//! Prior to sending a message, it is necessary to mint sufficient balance for
263//! the sender to ensure coverage of the existential deposit and gas costs.
264//!
265//! ```no_run
266//! # use gtest::constants::EXISTENTIAL_DEPOSIT;
267//! # let sys = gtest::System::new();
268//! let user_id = 42;
269//! sys.mint_to(user_id, EXISTENTIAL_DEPOSIT * 1000);
270//! ```
271//!
272//! Alternatively, you can use the default users from `gtest::constants`, which
273//! have a preallocated balance, as the message sender.
274//!
275//! ```no_run
276//! # use gtest::constants::DEFAULT_USERS_INITIAL_BALANCE;
277//! # let sys = gtest::System::new();
278//! assert_eq!(
279//!     sys.balance_of(gtest::constants::DEFAULT_USER_ALICE),
280//!     DEFAULT_USERS_INITIAL_BALANCE
281//! );
282//! ```
283//!
284//! ## Sending messages
285//!
286//! To send message to the program need to call one of two program's functions:
287//!
288//! - [`Program::send`] (or [`Program::send_with_value`] if you need to send a
289//!   message with attached funds).
290//! - [`Program::send_bytes`] (or [`Program::send_bytes_with_value`] if you need
291//!   to send a message with attached funds).
292//!
293//! Both of the methods require sender id as the first argument and the payload
294//! as second.
295//!
296//! The difference between them is pretty simple and similar to [`gstd`](https://docs.gear.rs/gstd/) functions [`msg::send`](https://docs.gear.rs/gstd/msg/fn.send.html) and [`msg::send_bytes`](https://docs.gear.rs/gstd/msg/fn.send_bytes.html).
297//!
298//! The first one requires payload to be CODEC Encodable, while the second
299//! requires payload implement `AsRef<[u8]>`, that means to be able to represent
300//! as bytes.
301//!
302//! [`Program::send`] uses [`Program::send_bytes`] under the hood with bytes
303//! from `payload.encode()`.
304//!
305//! First message to the initialized program structure is always the init
306//! message.
307//!
308//! ```no_run
309//! # let sys = gtest::System::new();
310//! # let prog = gtest::Program::current(&sys);
311//! let res = prog.send_bytes(100001, "INIT MESSAGE");
312//! ```
313//!
314//! ## Processing the result of the program execution
315//!
316//! Any sending functions in the lib returns an id of the sent message.
317//!
318//! In order to actually get the result of the program execution the block
319//! execution should be triggered (see "Block execution model" section).
320//! Block execution function returns the result of the block run
321//! ([`BlockRunResult`])
322//!
323//! It contains the final result of the processing message and others, which
324//! were created during the execution.
325//!
326//! It has 2 main functions:
327//!
328//! - [`BlockRunResult::log`] — returns the reference to the Vec produced to
329//!   users messages. You may assert them as you wish, iterating through them.
330//! - [`BlockRunResult::contains`] — returns bool which shows that logs contain
331//!   a given log. Syntax sugar around `res.log().iter().any(|v| v == arg)`.
332//!
333//! Fields of the type are public, and some of them can be really useful:
334//!
335//! - field `succeed` is a set of ids of messages that were successfully
336//!   executed.
337//! - field `failed` is a set of ids of messages that failed during the
338//!   execution.
339//!
340//! To build a log for assertion you need to use [`Log`] structure with its
341//! builders. All fields here are optional. Assertion with `Log`s from core are
342//! made on the `Some(..)` fields. You will run into panic if you try to set the
343//! already specified field.
344//!
345//! ```no_run
346//! # use gtest::Log;
347//! # use gear_core_errors::ErrorReplyReason;
348//! // Constructor for success log.
349//! let log = Log::builder();
350//!
351//! // Constructor for error reply log.
352//! let log = Log::error_builder(ErrorReplyReason::RemovedFromWaitlist);
353//! # let sys = gtest::System::new();
354//! # let prog = gtest::Program::current(&sys);
355//! // Other fields are set optionally by `dest()`, `source()`, `payload()`, `payload_bytes()`.
356//! let log = Log::builder()
357//!     .source(prog.id())
358//!     .dest(100001)
359//!     .payload_bytes("PONG");
360//! ```
361//!
362//! Log also has `From` implementations from `(ID, T)` and from `(ID_1, ID_2,
363//! T)`, where `ID: Into<ProgramIdWrapper>`, `T: AsRef<[u8]>`.
364//!
365//! ```no_run
366//! # use gtest::Log;
367//! let x = Log::builder().dest(5).payload_bytes("A");
368//! let x_from: Log = (5, "A").into();
369//! assert_eq!(x, x_from);
370//!
371//! let y = Log::builder().dest(5).source(15).payload_bytes("A");
372//! let y_from: Log = (15, 5, "A").into();
373//! assert_eq!(y, y_from);
374//! ```
375//!
376//! ## Blocks execution model
377//!
378//! Block execution has 2 main step:
379//! - tasks processing
380//! - messages processing
381//!
382//! Tasks processing is a step, when all scheduled for the current block number
383//! tasks are tried to be processed. This includes processing delayed
384//! dispatches, waking waited messages and etc.
385//!
386//! Messages processing is a step, when messages from the queue are processed
387//! until either the queue is empty or the block gas allowance is not enough for
388//! the execution.
389//!
390//! Blocks can't be "spent" without their execution except for use the
391//! [`System::run_scheduled_tasks`] method, which doesn't process the message
392//! queue, but only processes scheduled tasks triggering blocks info
393//! adjustments, which can be used to "spend" blocks.
394//!
395//! Note, that for now 1 block in Gear-based network is 3 sec duration.
396//!
397//! ```no_run
398//! # let sys = gtest::System::new();
399//! // Spend 150 blocks by running only the task pool (7.5 mins for 3 sec block).
400//! sys.run_scheduled_tasks(150);
401//! ```
402//!
403//! Note that processing messages (e.g. by using
404//! [`Program::send`]/[`Program::send_bytes`] methods) doesn't spend blocks, nor
405//! changes the timestamp. If you write time dependent logic, you should spend
406//! blocks manually.
407//!
408//! ## Balance
409//!
410//! There are certain invariants in `gtest` regarding user and program balances:
411//!
412//! * For a user to successfully send a message to the program, they must have
413//!   sufficient balance to cover the existential deposit and gas costs.
414//! * The program charges the existential deposit from the user upon receiving
415//!   the initial message.
416//!
417//! As previously mentioned [here](#Pre-requisites-for-Sending-a-Message),
418//! a balance for the user must be minted before sending a message. This balance
419//! should be sufficient to cover the following: the user's existential deposit,
420//! the existential deposit of the initialized program (the first message to the
421//! program charges the program's existential deposit from the sender), and the
422//! message's gas costs.
423//!
424//! The [`System::mint_to`] method can be utilized to allocate a balance to the
425//! user or the program. The [`System::balance_of`] method may be used to verify
426//! the current balance.
427//!
428//! ```no_run
429//! # use gtest::Program;
430//! # let sys = gtest::System::new();
431//! // If you need to send a message with value you have to mint balance for the message sender:
432//! let user_id = 42;
433//! sys.mint_to(user_id, 5000);
434//! assert_eq!(sys.balance_of(user_id), 5000);
435//!
436//! // To give the balance to the program you should use [`System::transfer`] method:
437//! let mut prog = Program::current(&sys);
438//! sys.transfer(user_id, prog.id(), 1000, true);
439//! assert_eq!(prog.balance(), 1000);
440//! ```
